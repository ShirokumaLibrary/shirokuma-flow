# 調査タスクの進め方

調査・分析タスク（ワークフロー調査、ギャップ分析、設計レビュー等）における作業の進め方を定義する。

## 1. サブエージェント丸投げの禁止

調査タスクではサブエージェントに丸投げせず、**マネージャー（メイン AI）自身がソースコードを読んで処理フロー詳細を書く**。

### 必ずやること

- 自分でソースコードを読む
- 「どのコマンドが・何を・どの順序で・どの関数を呼んで実行しているか」の具体的な処理フローを記述する
- エラー時の挙動・条件分岐・引数の流れも追う

### やってはいけないこと

- 「副作用の有無（○×）」の表だけで終わらせる
- ソースを読まずに副作用マトリクスをエージェントに作らせる
- 成果物をユーザーに見せる前に Issue や Discussion に投稿する
- コード変更がない調査タスクでブランチを作る

**過去の失敗:** サブエージェントに丸投げした結果、ソースを読まずに○×の副作用表だけが作られ、各コマンドの「何を、どの順序で、どの関数を呼んで実行しているか」が抜けた資料ができた。さらに結果を確認せずに Issue に 10 件コメント投稿し、全削除する羽目になった。

## 2. 表面的棚卸しではなく根本的な設計問題を検討する

調査タスクでは、関数呼び出しの棚卸しに終始せず、**現在の設計の前提を疑う**。

### 「理想設計」フェーズで問い直すこと

- このステータスモデル（FSM）でよいか
- この責務分割でよいか
- このパターンは本当に必要か
- 他コマンドとの一貫性はあるか
- 「理想設計」が現状の設計をそのまま FSM に書き下しただけになっていないか

**過去の失敗:** キャッシュ関数の呼び出し漏れを数える作業に時間を使い、以下の根本問題を見落とした:
- Spec Review ステータスがステータスモデルにないのに SKILL.md で参照されている
- PR が作成時に In Progress ステータスで始まる（Review であるべき）
- Issue と PR で同一ステータスモデルを共有しているがライフサイクルが異なる

## 3. レビューゲートで段階的に進む

調査成果物（Discussion、Issue 本文、レポート）を作成したら、**ユーザーレビューを挟んでから次のフェーズに進む**。

### フェーズごとの確認ポイント

| フェーズ | 確認ポイント |
|---------|------------|
| 処理フロー詳細作成 | この粒度・観点で十分か |
| 副作用マトリクス | 抽出した副作用の妥当性 |
| ギャップ分析 | 比較対象・基準の妥当性 |
| Issue 化 / ADR 化 | 実際に課題化する範囲 |

特に **ギャップ分析 → Issue 作成の前には必ずレビューを挟む**。一気に進めない。

**過去の失敗:** 浅い分析のまま D1〜D5 作成 → ギャップ分析 → 6 件の Issue 作成 → Status Review まで一気に進めた結果、的外れな Issue が作成された。ユーザーが根本問題を指摘して初めて気づいた。

## 4. レビュアーに確認すべきソースを渡さない

調査成果物の正確性・網羅性レビュー（漏れ検証レビュー）では、**レビュアーに「確認すべきソースファイル」の一覧を渡さない**。

### なぜか

漏れを検証するレビューなのに確認範囲を限定したら、漏れを検出できない。レビュアーには対象 Discussion / Issue の番号だけ渡し、自分でコードベース全体を探索させる。

### 渡すもの・渡さないもの

| カテゴリ | 渡す | 渡さない |
|---------|------|---------|
| 対象 | Discussion / Issue 番号 | — |
| 指示 | 「プロジェクト全体を探索して突き合わせろ」 | — |
| ヒント | — | 確認すべきソースファイル一覧 |
| 範囲 | — | ディレクトリ限定、探索のヒント |

**過去の失敗:** D1/D2 レビューで確認すべきソースを列挙して渡そうとしたが、漏れを検証するレビューなのに確認範囲を限定したら漏れを検出できない、と気づいた。
