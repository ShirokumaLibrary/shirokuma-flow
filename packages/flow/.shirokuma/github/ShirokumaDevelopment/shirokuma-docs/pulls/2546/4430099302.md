### Low 観察事項

#### [code] L1: `pushIssueBody` で非 Status フィールドが Status ガード前に書き込まれる（既存挙動の継続）

`packages/flow/src/commands/issue/push/issue.ts:359-393` の処理順序:

1. 行 361: 非 Status フィールド（Priority/Size）を `setItemFields` で更新
2. 行 371: ロールバック遷移ガード `validateTransition`
3. 行 381: Status を `updateProjectStatus` で更新

ローカルキャッシュが「Priority 変更 + ロールバック Status 変更」の両方を含み `--force` なしで実行された場合、Priority は GitHub に反映されるが Status はガードで `return 1`。**部分更新**が発生する。

これは方針 B 採用に伴う新規発生ではなく修正前から同じ挙動（修正前は `setItemFields` 内バリデーションが Status のみ拒否し Priority は既に書き込み済み）。リグレッションではないため本 PR のスコープ外で良い。

**対応案（任意・別 Issue）:** `pushIssueBody` 内で Status 変更を伴うロールバック遷移は事前にガードして全フィールド更新をスキップするよう順序入れ替えを検討。ただし「ロールバックも明示的に行いたい場合は `--force`」というポリシーが既に存在するため、現実装で許容可能。

### ベストプラクティス（良かった点）

- **方針 B の責務集約**: `setItemFields` を「フィールド更新の純粋なディスパッチャ」、ロールバックガードを「CLI 入口」に明確に分離。`updateProjectStatus` の `force` 削除も含め、シグネチャ縮小によりコードの責務境界が明確化。
- **JSDoc コメントによる方針記述**: `setItemFields`（行 368-369）と `updateProjectStatus`（行 296-297）の双方に「#2544: 二重バリデーション廃止」「ロールバックガードは各 CLI 入口で実施」と明記し、将来の実装者がガード位置を見失う risk を低減。
- **テストモック構造の刷新**: `push.test.ts` で従来の `mockSetItemFields.mockResolvedValueOnce(0)` ベースの間接検証から、`mockValidateTransition` 直接モックへの変換により、ガード位置移動に追従。
- **新規 `mockUpdateProjectStatus` の導入**: `expect(mockUpdateProjectStatus).not.toHaveBeenCalled()` で「ガード発動時に Status 更新が呼ばれない」ことを直接検証可能になり、テストの意図が明確化。
- **既存挙動の温存（不要な変更を避ける）**: `previousStatus` パラメータは `autoSetTimestamps` で引き続き必要なため `updateProjectStatus` から削除せず残した判断は適切。
