# コードレビュー: PR #2537 (sub1: CLI コア改修)

## サマリー

ADR-v3-022 第二改訂版（6 値モデル + 4 系統 STATUS_TRANSITIONS 分離）の起点として、`status-workflow.ts` の定数・関数を 6 値ベースに刷新し、新バリデーション API（`getAllowedTransitions` / `validateTransition`）を導入。テーブル定義は仕様 #2528 と一致しており、後方互換戦略（旧 API を `@deprecated` で残置）も適切。一方、`normalizeLegacyStatus` の `Approved → Done` 変更により integrity 系の既存ロジック（`classifyApprovedInconsistencies` の説明文・metadata と `--fix` の `approved-to-review` 動作の三者不整合）が破綻しており、`isBacklogEquivalent` の意味変更も使用箇所の意味付けがずれる懸念がある。テスト 2 ファイルの `.disabled` 化は status 関連以外の機能（cmdAddComment / cmdAddDiscussion / labelRules / IssueType / cache）も巻き込んでおり、sub4 まで広範なカバレッジ低下が残る。

## 重大な問題 (Critical / High)

### [Critical] `Approved` LEGACY の三者不整合（classify.ts と integrity index.ts の動作矛盾）

`packages/flow/src/utils/status-workflow.ts:122-123` で `normalizeLegacyStatus("Approved") → STATUS_VALUES.DONE` に変更されたが、関連 3 箇所が旧モデル（`Approved → Review`）の前提のまま残っている:

1. `packages/flow/src/commands/items/integrity/classify.ts:550-555` — `migrationTarget = normalizeLegacyStatus("Approved") = "Done"` が metadata に入る一方、description は `--fix で Review に戻します` と表示する。
2. `packages/flow/src/commands/items/integrity/index.ts:874-891` — `--fix` は `STATUS_VALUES.REVIEW` に更新し、`logger.success("Reverted #${item.number} status: Approved → Review")` を出す。`fixes.action` は `"approved-to-review"`。
3. `classify.ts:558` — description が `5 値モデル (ADR-v3-018) へ移行してください` のまま。6 値モデル（ADR-v3-022 第二改訂版）への移行を促すべき。

結果として、ユーザーが見るメッセージ・metadata・実際の遷移先がすべて食い違う。Approved を持つ既存 Issue に `--fix` を当てると、metadata は `Done` を示すのに実際は `Review` に遷移する。レビュー観点 5 で挙げられた「Approved を持つ既存 Issue の解釈を壊さないか」の懸念がここで顕在化している。

**推奨対応:**

- sub3 で integrity 系を 6 値対応する際にこの矛盾を解消する旨を本 PR コメントまたは Issue #2533 に明記する。
- 暫定で `classify.ts:558` の `5 値モデル (ADR-v3-018)` を `6 値モデル (ADR-v3-022 第二改訂版)` に修正する（コメント程度の変更で破壊リスクなし）。
- `classify.ts:555` の description を「`--fix` の挙動と新 migrationTarget が将来 sub3 で再整合される予定」と明示する。

### [High] `isBacklogEquivalent` の意味拡張による既存呼び出し側の意味ずれ

`packages/flow/src/utils/status-workflow.ts:181-188` で `isBacklogEquivalent` が `STATUS_VALUES.BACKLOG`（未調査）と `STATUS_VALUES.TODO`（着手準備完了）の両方を含むようになった。コメント (#2531) で意図的な選択と明記されているが、既存呼び出し側に意味のミスマッチが生じる:

- `packages/flow/src/commands/pr/merge.ts:425` — 「次の `Backlog` 計画」を選ぶフィルタに使用。コメント (`Pending / Ready 旧表記も Backlog 扱い`) は旧モデル前提。新モデルでは `ToDo` も含まれるため、計画 Issue が `ToDo` の場合も「次の Backlog 計画」として `implement-flow` が提案される。これは仕様上は妥当（着手準備完了 = 実装着手可能）だが、メッセージ `次の計画 #N が Backlog です` は誤表示になる（実際は `ToDo`）。
- `packages/flow/src/utils/parent-status.ts:389` — `allBacklog = subIssueStatuses.every((s) => isBacklogEquivalent(s))` で「全 Backlog ⇒ 親 ToDo」導出。新モデルでは「全サブが Backlog（未調査）」も「全サブが ToDo」も「Backlog と ToDo の混合」も全部 `STATUS_VALUES.TODO` に導出される。本来「全サブが Backlog（未調査）⇒ 親も Backlog」が自然だが、コードは ToDo を返す。

**推奨対応:**

- `pr/merge.ts:425` のフィルタは `isInvestigationPending` (Backlog 専用) または `isReadyForImplementation` (ToDo 専用) に置換する判断を sub3 で行う旨を計画に追加する。実装意図に応じて使い分けないと「次に着手すべき計画」の判定が曖昧になる。
- `parent-status.ts:389` の挙動を見直し、`isInvestigationPending` で全サブ Backlog のときは `STATUS_VALUES.BACKLOG` を返す分岐を追加する（sub3 スコープ）。

### [High] `INCOMPLETE_SUB_ISSUE_STATUSES` の `APPROVED_LEGACY` 残置のセマンティクス不整合

`packages/flow/src/utils/parent-status.ts:281-290` のコメントで `APPROVED_LEGACY` を「"Approved" → Done（旧 plan/design 承認済み、Done だが PR open 中の中間状態として未完了扱い継続）」としているが、新 6 値モデルでは `Approved` → `Done` であり、`Done` は完了扱い。一方 `INCOMPLETE_SUB_ISSUE_STATUSES` に `APPROVED_LEGACY` を残すことで、Approved を持つ既存サブ Issue は親 Done 連動 close を阻止する（hasIncompleteWorkSiblings）。これは意図通り保守的だが、コメント文と実挙動の説明（「Done だが…未完了扱い継続」）が論理的に分かりにくい。

**推奨対応:** sub3 で Approved を持つ既存 Issue を migrate-v3-018（または `--migrate-v3-022`）で Done にマイグレーションし、その後 `APPROVED_LEGACY` を `INCOMPLETE_SUB_ISSUE_STATUSES` から削除する戦略を sub3 / sub6 計画に明示する。

### [High] 旧 `STATUS_TRANSITIONS` の `Review → ToDo` が新モデルと矛盾する遷移を許容し続ける

`packages/flow/src/utils/status-workflow.ts:430-435` の旧 `STATUS_TRANSITIONS[Review]` は `[ToDo, Done, In progress]` を含む。`@deprecated` で残置するのは sub3 までの移行期間として妥当だが、新モデルでは `Review → ToDo` は廃止された遷移（`approve` の意味変更により `Review → Done` に統一）。

旧 API を呼び出す箇所:

- `packages/flow/src/commands/issue/close/index.ts:75`, `packages/flow/src/commands/issue/update/index.ts:409`, `packages/flow/src/commands/issue/reopen/index.ts:43`, `packages/flow/src/commands/status/transition/index.ts:250`, `packages/flow/src/commands/status/allowed.ts:37,70`, `packages/flow/src/commands/status/get.ts:41`, `packages/flow/src/utils/project-fields.ts:387`

これらが移行されるまで、CLI ユーザーは旧定義のまま `Review → ToDo` を実行可能。ADR-v3-022 第二改訂版で禁止された遷移が実質的に許容され続けるため、sub3 着手前に新モデルでの操作（特に `approve`）が一時的に旧仕様で動く期間が発生する。

**推奨対応:**

- 旧 `STATUS_TRANSITIONS` の `[STATUS_VALUES.REVIEW]` から `STATUS_VALUES.TODO` を削除する（残置するなら警告コメントを強化）。これは新仕様準拠であり、旧 API 呼び出し側に与える影響は `validateStatusTransition` が「非標準遷移」warning を返すのみなので破壊的ではない。
- もしくは sub2 でルーティング切替が完了した時点で sub3 を即座に着手する旨を計画に明記する。

## 改善点 (Medium)

### [Medium] `INITIAL_STATUSES = ["Backlog"]` 変更の影響を受けるコメント・既存スクリプトの未更新

`packages/flow/src/commands/issue/add/index.ts:162-164` のコメント `初期ステータスの検証（Backlog / Pending のみ許可）` は古い。`INITIAL_STATUSES` は `["Backlog"]` のみであり、`Pending` は `LEGACY_STATUS_VALUES.PENDING_LEGACY` で `normalizeLegacyStatus("Pending") → Backlog` だが `validateInitialStatus` はそのまま比較するため `Pending` 直指定は通らない。

**推奨対応:** コメントを `初期ステータスの検証（Backlog のみ許可）` に修正、または `validateInitialStatus` 内で `normalizeLegacyStatus(status)` を通してから比較する設計を検討（`Pending` 直指定の旧スクリプト互換のため）。

### [Medium] `STATUS_ORDER` への Backlog 追加で `isBackwardTransition` の意味が変わる

`packages/flow/src/utils/status-workflow.ts:149-156` で `Backlog` を index 0 に追加したことで:

- `Backlog → ToDo` は forward (0→1)、`Backlog → In progress` は forward (0→2)、`Backlog → Review` は forward (0→4) — すべて新モデルの `ISSUE_FORWARD_TRANSITIONS[Backlog]` と整合。
- `In progress → Backlog` は backward (2→0) — `ISSUE_ROLLBACK_TRANSITIONS[In progress]` と整合。
- `ToDo → Backlog` は backward (1→0) — ただし `ISSUE_ROLLBACK_TRANSITIONS[ToDo]` には Backlog が定義されていない。`isBackwardTransition` のみが使われる用途（`postStatusTransitionComment` の `(差し戻し)` ラベル付け、`autoSetTimestamps` の G12 タイムスタンプクリア）では「未定義の rollback でも backward 表示する」挙動になる。

これは実害はないが、4 系統テーブルと `STATUS_ORDER` の整合性に関する設計意図が ADR では明確でない。仕様 #2528 第二改訂版に沿うなら問題ないが、`ToDo → Backlog` を意図的に許容しない（rollback テーブル外）にも関わらず STATUS_ORDER 上は backward と判定されることを設計上明示してほしい。

**推奨対応:** `status-workflow.ts:149` のコメント `@since #2531` に「`STATUS_ORDER` は表示・メトリクスクリア用途であり、4 系統 transition テーブルとは独立した順序定義である」旨を追記する。

### [Medium] テスト 2 ファイルの `.disabled` 化により status 以外の広範な機能カバレッジが消失

`packages/flow/__tests__/commands/issue/add.legacy.ts.disabled` は `cmdAddIssue` 33 テスト中の以下を含む:

- `cmdAddComment` 6 テスト（Issue/PR/Discussion 自動判別）
- `cmdAddIssue` の labelRules / IssueType 解決 / assignees / cache 系 約 17 テスト
- `cmdAddDiscussion` 4 テスト

これらの大半は status 6 値化と無関係な機能であり、sub4 完了まで（恐らく数日〜数週間）カバレッジ低下が残る。`status が In progress の場合は正常に Issue を作成する` のような status 直接関連テストのみ修正し、その他のテストは維持できないか。

**推奨対応:**

- 短期: `add.legacy.ts.disabled` のうち status 検証 5 テスト（`status が Review`, `status が In progress`, `status が Pending`, `status 省略 + getDefaultStatus()`）のみ disabled に分離し、残り 28 テストを `.test.ts` に戻す。
- これが過大なら、sub4 のスコープに「status 非依存テスト 28 件の即時復活」を最優先タスクとして明記する。

### [Medium] `gh-config.ts:14` で `LEGACY_STATUS_VALUES` を import するが新モデルでも `IN_PROGRESS_LEGACY` のみ使用

`packages/flow/src/utils/gh-config.ts:64-72` の `DEFAULT_METRICS.statusToDateMapping` で `LEGACY_STATUS_VALUES.IN_PROGRESS_LEGACY: "Start at"` を使う。これは GitHub Project の旧 option 名「In Progress」（大文字 P）に揃えるためで、6 値モデル化とは独立した設計。問題ではないが、`gh-config.ts:11` の import が `LEGACY_STATUS_VALUES` を引き続き必要とすることをコメントで補足するとよい。

**推奨対応:** `gh-config.ts:62-63` のコメント「マッピングキーは GitHub Project の Status field option 名と一致させる必要がある。既存 Project は旧表記 "In Progress"（大文字 P）のまま」は既に明確。問題なし。（取り下げ）

### [Low] `LEGACY_STATUS_VALUES` 定義行 73-78 のコメント「削除済みステータス」が ICEBOX/PREPARING/DESIGNING/TESTING/RELEASED に対する説明として簡潔すぎる

### [Low] テスト `gh-config.test.ts:171` の `#2439` コメント残置

## ベストプラクティス

- 4 テーブル分離（`ISSUE_FORWARD_TRANSITIONS` / `ISSUE_ROLLBACK_TRANSITIONS` / `PR_FORWARD_TRANSITIONS` / `PR_ROLLBACK_TRANSITIONS`）の定義はテーブル形式 JSDoc コメント付きで非常に読みやすく、仕様 #2528 第二改訂版との照合が容易。
- `getAllowedTransitions(itemType, from, opts)` / `validateTransition(itemType, from, to, opts)` のシグネチャは `itemType` 必須かつ `opts.allowRollback` でフラグ化されており、レビュー観点 2 の妥当性チェックは PASS。`{ valid, kind: "forward" | "rollback", warning, error }` の戻り値も呼び出し側で扱いやすい。
- `--rollback` フラグなしで rollback 先を指定したケースに専用エラー（`status-workflow.ts:540-545`）で誘導する実装は UX 上良い。
- `isInvestigationPending` / `isReadyForImplementation` の細分化判定の追加は、`isBacklogEquivalent` のセマンティクス曖昧化の補完策として適切。
- 後方互換戦略（旧 `STATUS_TRANSITIONS` / `validateStatusTransition` を `@deprecated` で残し sub3 で削除）は段階的移行として理にかなっている。
- 旧 API の JSDoc に「sub3 で全呼び出し側を移行後に削除する」と明示している点は、レビュー観点 3 で求められた移行期間の問題回避に貢献。
