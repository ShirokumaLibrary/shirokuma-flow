## ベストプラクティス

### 良い点

- **4 系統テーブル分離**: `ISSUE_FORWARD / ISSUE_ROLLBACK / PR_FORWARD / PR_ROLLBACK` の責務分離が明快で、`getAllowedTransitions(itemType, from, opts)` の API 設計も堅実。型 (`itemType: "issue" | "pr"`) で誤用を防いでいる。
- **`--rollback` 必須化**: `validateTransition` の専用エラーメッセージ（`--rollback フラグが必要です`）と `reject` の二重チェック（`requiredOption` + ランタイム check）でユーザー誤用を多重防御している。
- **互換層の正規化**: `normalizeLegacyStatus` で `Pending → Backlog`, `Approved → Done` を一元化し、`isInvestigationPending` / `isReadyForImplementation` の細分化判定を `isBacklogEquivalent` から分離している点は将来の保守性が高い。
- **`syncParentStatus` の最大深さガード（MAX_PARENT_SYNC_DEPTH = 2）と訪問済みセット**: 再帰的な親伝播のループ防止が適切。
- **既存 Issue 移行をしない方針の徹底**: `migrate-v3-018` 系コードを sub6 で削除し、過渡期は LEGACY 透過層で吸収する戦略が一貫している。
