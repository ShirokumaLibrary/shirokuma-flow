## 追記: キャッシュ一貫性バグを本 PR で修正

上記サマリーの「補足」で別 Issue 化を推奨した Status キャッシュ一貫性バグは、ご指示により**本 PR #2685 内で修正しました**（commit `c960a135`）。

**根本原因:** `status get` / `begin` / `status transition` / `submit` / `reject` が使う `resolveCurrentStatus`（および `transition` の重複 `getCurrentStatus`）は `.shirokuma/cache/issues/{n}.json` を**鮮度チェックなし・バイパス手段なし**でキャッシュ優先読みしていた。`issue context` は `--no-cache` でしか再取得せず（`--refresh` フラグは存在しなかった）、`issue pull` は `body.md` のみ更新するため、外部で Status が変わるとローカルキャッシュが古いまま固定され `begin` が遷移を誤拒否した。

**修正:**
- `resolveCurrentStatus` に `refresh` オプションを追加（キャッシュをバイパスしてライブ取得 → 成功時のみキャッシュへ書き戻して sticky 化。失敗・null・PR フォールバック時は既存値を保護）。
- `status get` / `status allowed` / `status transition` / `submit` / `reject` / `begin` / `issue context` に `--refresh` フラグを追加（`issue context --refresh` は `--no-cache` エイリアス）。
- `transition` の重複 `getCurrentStatus` を `resolveCurrentStatus` に一本化（約 106 行削減）。
- 再現テスト・refresh 経路テストを追加。

今後は `begin 2684 --refresh` / `issue context 2684 --refresh` で手動キャッシュ書き換えなしにライブ Status へ復帰できます。
