## 計画レビュー結果

**レビュー結果:** NEEDS_REVISION

`syncChildCloseOnParentClose` の `cascadeCloseTargets` フィルタを最小拡張する方針自体は既存実装と整合的で筋が良い。ただし「3 トリガーで自動的にカバー」という主張は `issue rollback --action cancel` 経路と `pr merge` 経由の連鎖モデルで成り立たない部分があり、また「実作業サブ Issue」の定義は `isPlanOrDesignIssue` の判定基準（ラベル優先・タイトルプレフィックスはフォールバック）の運用前提と、Done/Approved ステータス以外の Open 子（Backlog/ToDo/In Progress/Review/Blocked）まですべて連動 Close することのリスク評価が計画から抜けている。

### 重大な問題

#### [Critical][計画] 「3 トリガーで自動カバー」の主張に経路漏れがある（検討事項 1）

計画では `issue close` / `issue cancel` / `pr merge` の 3 トリガーが既存 `syncChildCloseOnParentClose` 呼び出しで自動カバーされると主張しているが、実装を確認すると以下の不整合がある。

1. **`issue cancel` には 2 経路ある**
   - `cmd issue cancel`（`packages/flow/src/commands/cancel.ts` → `cmdItemClose` 経由）は `syncChildCloseOnParentClose` を呼ぶので OK。
   - `issue rollback --action cancel`（`packages/flow/src/commands/issue/rollback/index.ts:397`）は `closeIssueById(issueId, "NOT_PLANNED")` を**直接呼び出すだけ**で、`syncChildCloseOnParentClose` は経由しない。この経路は逆方向連鎖の対象外なのか、それともカバーするのかが計画に明記されていない。
   - 親 Issue の close 手段として rollback の cancel が使われた場合、計画が想定する逆方向連鎖が発火しないため、ユーザー体感としての一貫性が壊れる。

2. **`pr merge` 経由の「親 close → 子 close」連鎖は現状の実装では発生しない**
   - `pr/merge.ts:243` の `syncChildCloseOnParentClose(owner, repo, num, logger)` は **PR にリンクされた `num`（通常は実作業サブ Issue）の子** に対する連動 close。実作業サブ Issue は通常子を持たないため、ここでの連鎖は空に近い。
   - PR マージで「親（計画 Issue / エピック）が `syncParentStatus` 経由で Done + Closed に自動遷移したあと、その親の他の子も連動 close する」連鎖は、`syncParentStatus`／`closeDoneParent` が **`syncChildCloseOnParentClose` を呼び出していない**ため発火しない（`parent-status.ts:461-490` 参照）。
   - したがって「PR マージで親が自動 close → 兄弟子も自動 close」は現状未実装。計画が「`pr merge` 経由は既存呼び出しで自動的にカバーされる」と主張するなら、**何が自動カバーされて何がされないか**を明示する必要がある。

**修正提案:**
- 計画内に「対象スコープ」を明記する。`issue rollback --action cancel` を含めるなら同じファイルに `syncChildCloseOnParentClose` 呼び出しを追加するタスクが必要。含めないなら明示的にスコープ外宣言する。
- `pr merge` 経由の連鎖を主張するなら、`syncParentStatus`（または `closeDoneParent`）から `syncChildCloseOnParentClose` を呼ぶ追加変更が必要かを評価する。本当に「人が `issue close` した時にだけ発火する逆方向連鎖」が要件なら、計画の主張から「pr merge も自動カバー」の文言を削除すべき。

### 改善点

#### [High][計画] フィルタ拡張の対象範囲が「Open 状態の実作業サブ Issue 全て」になり安全側にすぎる

計画では「`cascadeCloseTargets` フィルタに『計画 Issue / 設計 Issue でない Open 子』を追加する」と書かれているが、これだと **Backlog / ToDo / In Progress / Review / Blocked のすべてのオープン状態** が連動 Close 対象になる。

- 既存ロジック (`parent-status.ts:787-804`) は Done(Open) / Approved(Open) のみを連動 Close 対象にしており「完了済みだが Open で残っている子をクリーンアップする」という意味論。
- 「親が close されたなら未完了の子も含めて閉じる」という意味論への**質的拡張**になるが、その正当性（なぜ In Progress の子も自動 close してよいか）が計画に書かれていない。
- 親 Issue #2502 の AC は「実作業サブ Issue が自動 close される」としているが、未着手・進行中の子まで含むのかは明示されていない。**要件側のサブ Issue の状態についての扱いを確認した上で、計画でも明文化する必要がある。**

**修正提案:**
- 連動 Close 対象を「Open 状態のすべての実作業サブ Issue」とするのか「特定ステータスのみ」とするのかの判断根拠を計画に明記する。
- 進行中の子を `NOT_PLANNED` でなく `COMPLETED` で閉じることに違和感がないか（親 close 理由を継承する設計が常に妥当か）も再確認する。

#### [High][計画] `pr merge` 呼び出し時の `parentStateReason` 引数が省略されている問題

`pr/merge.ts:243` の `syncChildCloseOnParentClose(owner, repo, num, logger)` は `parentStateReason` 引数を渡しておらず、デフォルト `COMPLETED` で動作する。これは現行仕様としては合理的だが、**実作業サブ Issue まで連動 close 対象を広げる場合、PR マージ起因の cascade で進行中の子が `COMPLETED` で閉じられて履歴に残る** ことになる。

- 既存実装の `parentStateReason` 伝搬テスト（`parent-status-child-close.test.ts:236-270`）は cmdItemClose 経由の伝搬しかカバーしていない。
- 計画は「`parentStateReason` 伝搬は追加変更なし」と書いているが、対象範囲拡張後にこの引数省略が問題を起こさないかの評価が必要。

**修正提案:**
- `pr merge` 経由で実作業サブ Issue まで連動 close するシナリオが本当に必要か、必要なら `parentStateReason` をどう決めるかを計画に追記する。
- 単純な解は「`pr merge` 経由では実作業サブ Issue まで cascade しない（フラグで分岐）」だが、その場合「3 トリガーで自動カバー」の主張は再修正が必要。

#### [High][計画] テスト追加範囲に「経路別の挙動差」のシナリオが入っていない

計画のタスク分解にあるテストは「親 close 時に実作業サブが close される」「PR/孫がある場合は警告のみ」の 2 種類だが、上記の経路別問題（rollback cancel 未経由、pr merge 経由での parentStateReason 引数省略）の検証シナリオが含まれていない。

**修正提案（タスク分解への追加）:**
- `pr/merge.ts` から `syncChildCloseOnParentClose` が呼ばれた場合の cascade 挙動の単体テスト追加（pr/merge.test.ts または新規）
- `issue rollback --action cancel` 経路で逆方向連鎖が発火するか / しないかを期待値として明示するテスト

#### [Medium][計画] 「子に PR/孫 Issue がある場合」のテストが内容として不十分

タスク分解 4 番「子に PR/孫 Issue がある場合は警告のみで継続する」と書かれているが、計画文中は「PR がある場合: 警告ログのみで close する」「孫がある場合: close しない（1 段のみ）」と異なる挙動を述べている。一方タスク分解では「PR 有無は現行ロジックで変わらないため、ログ呼び出し確認程度」と緩いトーン。

`syncChildCloseOnParentClose` のフィルタは PR/孫の有無を判定する処理を**現状持たない**ため、計画通りなら「子に PR/孫があっても無差別に close する」のが実装結果となる。

**修正提案:**
- 「子に PR/孫がある場合の挙動」を「制限を設けず close（警告も出さない）」とするのか「警告を出して継続」とするのかを明確化。後者の場合は実装タスクが追加される（現状無し）。
- 孫 Issue の close 防止は「`syncChildCloseOnParentClose` が再帰しない」ことだけが保証する点を計画に明記し、テストでも「子の close が孫まで到達しない」ことを明示的に検証する。

#### [Medium][計画] 計画/設計 Issue のタイトル prefix 判定は既存ヘルパーで担保されるが、混在ケースの言及が無い

質問の重点 (2) について: 既存の `isPlanOrDesignIssue(node)` は「ラベル `area:plan`/`area:design` を正、タイトル prefix `計画:`/`Plan:`/`設計:`/`Design:` をフォールバック」で判定する（`parent-status.ts:212-268`）。設計 Issue は現時点で `area:design` ラベル未運用とコメントにある（`parent-status.ts:241`）。これにより、

- 実作業 Issue タイトルが「計画レビューを実装」のように「計画」を含むだけでは `startsWith` で除外されないので誤判定は起きにくい。
- ただし「Plan:」「Design:」始まりの実作業 Issue が誤って「計画/設計 Issue」とみなされる可能性は残る（運用上はないが）。

計画ドキュメント上、`isPlanOrDesignIssue` を採用する旨は明記されているが、「タイトル prefix 単独で判定する場合に誤判定リスクがあるが、現行運用では発生しないと判断した」という言及があるとレビュー時の理解が早い。

**修正提案:** 計画に「`isPlanOrDesignIssue` の判定基準（ラベル優先、タイトル prefix フォールバック）に依拠する。設計 Issue は `area:design` ラベル未運用のためタイトル prefix が事実上の判定根拠」を一文追記する。

#### [Medium][計画] エピック親（実作業 Issue を子に持つトップレベル親）と中間親で挙動が変わらないかの評価

逆方向連鎖は「親が close されたら子を close」だが、リポジトリでは以下の階層構造がある:

- **エピック親 #2502**（実作業 Issue を直接子に持つ）
- **計画 Issue #2547**（タスク Issue を子に持つ）
- 実作業サブ Issue（子を持たないことが多い）

計画では「親 → 子の一方向 close で、子の close が祖父 Issue に向かって親ステータスを更新することはあっても、祖父から再び孫を close する経路は生じない（逆方向連鎖は 1 段のみ）」と書いてあるが、これは「`syncChildCloseOnParentClose` 内で再帰呼び出しを行わない」という意味と読める。しかし子の close は `syncParentStatus` 側で親方向に伝搬する（`closeDoneParent` 経由）ため、**祖父も Done + Closed に遷移する可能性**がある。

- 実装としては問題ない（祖父の close で `syncChildCloseOnParentClose` は呼ばれないため）。
- ただし「逆方向 1 段」と書きながら「子 close → 親 close（正方向の `syncParentStatus`）→ 親 close で `syncChildCloseOnParentClose` は再発火しない」というロジック構造を計画にもう少し明確に書くと、レビュー・実装時の認識ずれを防げる。

**修正提案:** 計画の「ループ防止」項目で「`closeIssueById`（子に対する close）は `syncParentStatus` を呼ばないこと、`closeDoneParent` も `syncChildCloseOnParentClose` を呼ばないこと」を構造的に確認した旨を一文追記。

#### [Low][計画] エピック親 #2502 自身が close される時のシナリオが計画から見えにくい

計画タイトルは「親 Issue close 時に『計画 Issue 以外の子』を自動 close」だが、対象としている「親」の階層が明示されていない。エピック親（#2502 のような）と、計画 Issue（#2547 のような）と、実作業 Issue では子の構成が異なる。

- エピック親 close → 計画 Issue（Open）と実作業サブ Issue（Open）の両方を close（既存 + 新規拡張で両方カバー）
- 計画 Issue close → 実作業サブ Issue（Open）を close（新規拡張部分）

**修正提案:** 計画文の冒頭に「対象範囲: エピック・計画 Issue を含めた全ての『親』に対して、Open 状態の実作業サブ Issue を連動 close 対象に追加する」と書く。

#### [Low][計画] テストの正方向連鎖リグレッション検証範囲

タスク分解の 5 番「`parent-status.test.ts` / `sync-parent-status.test.ts` で正方向連鎖のリグレッションを確認」は妥当だが、これは「壊れる場合は修正する」のような曖昧な書き方になっている。フィルタ拡張は `syncChildCloseOnParentClose` のみで `syncParentStatus` には触れないため、**正方向連鎖が壊れる根拠は乏しい**。テスト確認が「既存テストが PASS することの確認だけ」で良いなら、その旨を明示する方が実装者が迷わない。

**修正提案:** 「既存テストが PASS することを確認するのみ（コードに変更を加える可能性は低い）」のニュアンスでタスク分解を書き直す。

### 推奨事項

1. **スコープ宣言の明確化（Critical 解消）**: 計画文に「対象トリガー」と「対象外トリガー（理由付き）」をリスト化する。`pr merge` 経由の自動カバーが本当に成立するのか、`issue rollback --action cancel` を含めるのかを明示する。
2. **対象子の状態の決定根拠を明記（High 解消）**: Open すべてを対象とするのか、特定ステータスのみかを明文化。親 Issue #2502 の AC 「実作業サブ Issue が自動 close される」を本計画レベルで Open 状態のすべての実作業子と読み替えて良いか、要件確認も推奨。
3. **`pr merge` 経由 cascade の `parentStateReason` の扱い**: `COMPLETED` 固定でよいか、新たに引数追加が必要かを評価。
4. **追加テストシナリオ**: 経路別（rollback cancel 経由 / pr merge 経由）の挙動を明示するテストを追加。
5. **再帰しないことの構造的説明**: 「`closeDoneParent` は `syncChildCloseOnParentClose` を呼ばない」「`syncChildCloseOnParentClose` は子の close 後に再帰しない」を計画に書き、ループ非発生を構造で示す。
6. **PR/孫の挙動の言葉統一**: 「警告のみで継続」「制限なく close」のどちらかに統一し、必要なら実装タスクを追加する。
