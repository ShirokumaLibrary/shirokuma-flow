## 計画の判断根拠

### 選定アプローチ

`syncChildCloseOnParentClose()` の `cascadeCloseTargets` フィルタ条件を拡張する。現在このフィルタは「Done(Open) またはオープン状態の計画/設計 Issue」のみを対象としているが、ここに「計画/設計 Issue でないオープン状態の子（実作業サブ Issue）」を追加する。コマンドレベル（`cmdItemClose`、`pr/merge.ts`）はすでに `syncChildCloseOnParentClose` を呼び出しており、追加のコマンド変更は不要。1 関数・1 ファイルの変更で全トリガー（`issue close`・`issue cancel`・`pr merge`）を網羅できる。

### 検討した代替案

**新規関数 `cascadeCloseWorkChildren()` を追加する案**

既存 `syncChildCloseOnParentClose` に手を加えず、実作業子専用のクローズ関数を別途実装し、`cmdItemClose` から個別に呼び出す案。しかし既存関数がすでに「計画/設計 Issue を Open 状態で連動 Close する」ロジックを持ち、`parentStateReason` 伝搬も済んでいるため、実質同じ処理が二重実装になる。却下。

**`cmdItemClose` 側で子を close する案**

コマンドハンドラ（`cmdItemClose`）に子 Issue クローズ処理を追加する案。ただし `pr merge` 経由（`pr/merge.ts`）で同様の処理が必要になり、2 箇所に分散する。却下。

### 調査で判明した制約

- `syncChildCloseOnParentClose` 内で `isPlanIssue(n)` の判定は行われているが、`isDesignIssue` / `isPlanOrDesignIssue` は現在フィルタ条件に含まれていない。設計 Issue もスキップしたい場合は `isPlanOrDesignIssue(n)` を使うか、`isPlanIssue(n)` のみのままにするか判断が必要。既存コードで `isPlanOrDesignIssue` が export されており、設計 Issue も除外するのが自然な設計のため `isPlanOrDesignIssue` を採用する。
- `syncParentStatus` の再帰ガード（`visited` セット + `MAX_PARENT_SYNC_DEPTH = 2`）は親方向連鎖を対象としており、子方向連鎖との相互再帰は構造上発生しない。追加ガードは不要。
- 子 Issue の close により `syncParentStatus` が再発動して親が Done → Closed になる可能性があるが、これは想定内の正常動作（親 close 後に兄弟を close すると祖父が Done になる）。ループにはならない。
