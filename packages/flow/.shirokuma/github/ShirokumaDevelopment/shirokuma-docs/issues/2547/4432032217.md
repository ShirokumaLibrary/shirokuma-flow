## 計画修正の判断根拠（NEEDS_REVISION 対応）

### Critical 対応: 3 トリガーの経路分析と修正方針確定

レビューで指摘された「3 トリガーで自動カバー」の主張が成立しない経路を実装確認し、以下の修正方針を確定した。

**`issue rollback --action cancel`（方針 B 採用）**

`rollback/index.ts:397` は `closeIssueById(issueId, "NOT_PLANNED")` を直呼びしており `syncChildCloseOnParentClose` を経由しない。この経路でも逆方向連鎖を発火させるため、`closeIssueById` 呼び出し直前に `syncChildCloseOnParentClose(owner, repo, number, logger, "NOT_PLANNED")` を追加する（方針 B: 親 close エントリから直接呼び出す）。

**`pr merge` 経由（スコープ外宣言）**

`pr/merge.ts:243` の `syncChildCloseOnParentClose` は PR にリンクされた実作業 Issue（`num`）の子に呼ばれる。実作業 Issue は通常子を持たないため連鎖は空。また「PR マージ → `syncParentStatus` → `closeDoneParent`」のルートも `syncChildCloseOnParentClose` を呼ばない（`parent-status.ts:461-490`）。「兄弟子の cascade」は現状未実装であり、本 Issue のスコープ外と明示する。

### H1 対応: Open 状態全体を対象とする正当性

既存フィルタの意味論（「完了済みだが Open で残っている子のクリーンアップ」）から「未完了の子も含めて閉じる」への質的拡張であることを計画に明文化した。親 close/cancel は当該タスクの中止・完了を意味するため、未完了の実作業を継続するユーザー意図は想定しないという設計判断。

### H2 対応: 経路別テスト追加

- `rollback.test.ts` に rollback cancel 経由の逆方向連鎖発火シナリオを追加
- `parent-status-child-close.test.ts` に孫非到達・`pr merge` 経由スコープ外の明示シナリオを追加

### H3 / M1 対応: isPlanOrDesignIssue の判定基準明文化

`isPlanOrDesignIssue(n)` はラベル（`area:plan`/`area:design`）優先、タイトル prefix（`計画:`/`Plan:`/`設計:`/`Design:`）フォールバック。設計 Issue は `area:design` 未運用のためタイトル prefix が事実上の判定根拠。誤判定リスクは運用上ない。

### M2 対応: 相互再帰回避の構造的保証

`closeIssueById`（子に対する close）は `syncParentStatus` を呼ばない。`closeDoneParent` は `syncChildCloseOnParentClose` を呼ばない。この 2 点により「逆方向 → 正方向 → 逆方向」の相互再帰は構造上発生しない。

### M3 対応: 子に PR/孫がある場合の挙動統一

PR 有無の判定は行わない（無差別に close）。孫は `syncChildCloseOnParentClose` が再帰しないため到達しない（1 段保証）。「警告のみで継続」の記述を削除し「制限なし close、孫は非到達」に統一した。
